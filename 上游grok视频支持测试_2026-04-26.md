# Grok 上游视频支持测试

测试日期：2026-04-26  
文档版本：v1.2

最新复测结论：见第 10 节；本轮上游第 3 次 `/v1/videos` 已成功返回 `status=completed` 和 `video/mp4` 文件片段，`sub-lb.tap365.org` 公网 `/v1/videos` 仍为 `404`。

## 1. 测试目标
验证 `https://grok.tap365.org` 这个上游本身是否暴露视频相关 API，以及两把用户提供的 `sub-lb/grok图文/SK(脱敏)` 是否能直接在上游完成视频相关业务调用。

## 2. 测试类型
业务接口实测 + 路由清单核对

## 3. 测试环境
- 仓库：`/Users/houzi/code/06-production-business-money-live/sublb-demo`
- 机器：本机 macOS / Codex Shell
- 目标 Host：`https://grok.tap365.org`
- 认证方式：
  - 第一把：`sub-lb/grok图文/SK(脱敏-1)`，仅通过 `Authorization: Bearer` 使用
  - 第二把：`sub-lb/grok图文/SK(脱敏-2)`，仅通过 `Authorization: Bearer` 使用
  - 文档中不记录明文 key

## 4. 测试对象
- `GET /openapi.json`
- `GET /v1/models`
- `POST /v1/chat/completions`
- `POST /v1/videos`
- `POST /v1/function/video/start`
- `GET /sora/v1/models`
- `POST /sora/v1/chat/completions`
- `GET /api/v1/sora/models`
- `POST /api/v1/sora/generate`
- `POST /v1/videos/generations`

## 5. 执行步骤
1. 先请求 `GET https://grok.tap365.org/openapi.json`，确认上游真实暴露的接口清单。
2. 先用第一把 key 直打 `GET /v1/models` 与 `POST /v1/chat/completions`，验证这把 key 是否被上游直接接受。
3. 在第一把 key 失败后，改用第二把 key 再次直打 `GET /v1/models` 与 `POST /v1/chat/completions`，确认是否能通过上游认证。
4. 再用第二把 key 直打真实视频入口：
   - `POST /v1/videos`
   - `POST /v1/function/video/start`
5. 再分别测试上游常见但可能错误的视频路径：
   - `/sora/v1/models`
   - `/sora/v1/chat/completions`
   - `/api/v1/sora/models`
   - `/api/v1/sora/generate`
   - `/v1/videos/generations`
6. 从 `openapi.json` 里额外提取 `video/sora/model` 相关路径，核对“上游到底宣告了哪些视频接口”。

## 6. 实际结果

### 6.1 上游路由真相
`GET /openapi.json` 返回 `200 application/json`，上游 OpenAPI 标题为 `Grok2API`，其中可见与视频相关的真实路径包括：
- `/v1/videos`
- `/v1/video/extend`
- `/v1/function/video/start`
- `/v1/function/video/stop`
- `/v1/function/video/sse`
- `/v1/files/video/{filename}`

说明：上游代码/服务层面**存在视频相关接口族**，但不是此前 sublb 外层尝试的 `/sora/*` 路由。

### 6.2 第一把 key 直打上游认证结果
- `GET /v1/models` → `401`
- `POST /v1/chat/completions` → `401`
- 返回体一致为：`{"error":{"message":"Invalid authentication token","type":"authentication_error","code":"invalid_api_key"}}`

说明：第一把 key 不是 grok.tap365.org 上游可直接接受的 key。

### 6.3 第二把 key 直打上游认证结果
- `GET /v1/models` → `200`
- 返回模型列表包含：
  - `grok-3`
  - `grok-4`
  - `grok-imagine-1.0`
  - `grok-imagine-1.0-edit`
  - `grok-imagine-1.0-video`
- `POST /v1/chat/completions` → `200`
- 返回为 SSE 流，正文出现 `ok`

说明：第二把 key 可以被 `grok.tap365.org` 上游直接接受，文本基线已通过。

### 6.4 第二把 key 的真实视频接口结果
- `POST /v1/videos`（`seconds=3`）→ `400`
  - 返回：`seconds must be between 6 and 30`
  - 说明：该接口不是空壳，已进入真实参数校验逻辑。
- `POST /v1/videos`（`seconds=6`）→ `502`
  - 返回：`AppChatReverse: Chat failed, 429`
  - 说明：请求已经进入上游视频生成链路，但被更内层 upstream/rate limit 拒绝，不是“接口不存在”。
- `POST /v1/function/video/start` → `401`
  - 返回：`Function access is disabled`
  - 说明：这把 key 可用于 `/v1/videos`，但**不具备 function 路由访问权限**。

### 6.5 之前猜测的 `/sora/*` 路径在上游结果
- `GET /sora/v1/models` → `404 Not Found`
- `POST /sora/v1/chat/completions` → `404 Not Found`
- `GET /api/v1/sora/models` → `404 Not Found`
- `POST /api/v1/sora/generate` → `404 Not Found`
- `POST /v1/videos/generations` → `404 Not Found`

说明：这批路径**不是当前 grok.tap365.org 上游的真实公开视频入口**。

## 7. 结论
结论：**上游 `grok.tap365.org` 确实支持视频生成主入口 `/v1/videos`，并且第二把 key 已通过上游文本认证；但当前视频实测被更内层 upstream `429` 限流拦住，`/v1/function/video/start` 这条 function 路由则被权限拦截。**

更准确地说：
1. `grok.tap365.org` 上游本身**不是完全没有视频能力**，因为 `openapi.json` 已声明一组 `/v1/videos` / `/v1/function/video/*` 路由。
2. 第一把 key 无法直连上游完成认证，但第二把 key 可以。
3. `POST /v1/videos` 对非法 `seconds` 返回真实业务校验错误，对合法参数则继续打到内层链路并返回 `AppChatReverse: Chat failed, 429`，说明**视频入口真实存在且已连到上游生成链路**。
4. 本轮仍**未拿到成功创建并可轮询完成的视频任务结果**，因此“支持视频入口”与“当前可稳定产出视频”要分开表述。
5. 现在如果要测 sublb 外层视频生成，已经有意义；但预期风险不是“上游没接口”，而是**内层 upstream 429 / function 路由权限受限**。

## 8. 失败信息

## 10. 二次复测：上游视频已成功，sub-lb 公网入口仍未暴露

### 10.1 测试日期与版本
- 测试日期：2026-04-26
- 文档版本：v1.2

### 10.2 测试目标
在用户完成上游侧调整后，重新验证：
1. `https://grok.tap365.org/v1/videos` 是否能真实产出视频。
2. 上游成功后，`https://sub-lb.tap365.org/v1/videos` 是否已经能通过公网 sublb 转发。

### 10.3 测试类型
业务接口实测。

### 10.4 测试环境
- 仓库：`/Users/houzi/code/06-production-business-money-live/sublb-demo`
- 机器：本机 macOS / Codex Shell
- 上游目标 Host：`https://grok.tap365.org`
- sublb 目标 Host：`https://sub-lb.tap365.org`
- 认证方式：
  - 上游：`grok.tap365.org/upstream/SK(脱敏)`，仅通过 `Authorization: Bearer` 使用
  - sublb：`sub-lb.tap365.org/grok图文/SK(脱敏)`，仅通过 `Authorization: Bearer` 使用
  - 文档与证据不记录明文 key

### 10.5 测试对象
- `GET https://grok.tap365.org/v1/models`
- `POST https://grok.tap365.org/v1/videos`
- `GET https://grok.tap365.org/v1/files/video/.../generated_video.mp4`（Range 方式校验真实视频文件）
- `GET https://sub-lb.tap365.org/v1/models`
- `POST https://sub-lb.tap365.org/v1/videos`

### 10.6 执行步骤
1. 使用上游 key 调用 `GET /v1/models`，确认认证与模型清单。
2. 使用相同上游 key 连续三次调用 `POST /v1/videos`，参数为：
   - `model=grok-imagine-1.0-video`
   - `size=1792x1024`
   - `seconds=6`
   - `quality=standard`
3. 第三次上游返回视频 URL 后，对该 URL 发起 Range 请求读取前 2048 字节，校验文件类型与 HTTP 状态。
4. 使用 sublb 的 grok 图文 key 调用 `GET /v1/models` 和 `POST /v1/videos`，确认公网 sublb 层是否已经暴露视频入口。

### 10.7 实际结果

#### 上游 `grok.tap365.org`
- `GET /v1/models` → HTTP `200`，返回 `data` 数量为 `15`，模型清单仍包含视频模型。
- `POST /v1/videos` 第 1 次 → HTTP `502`
  - 返回：`AppChatReverse: Chat failed, 429`
- `POST /v1/videos` 第 2 次 → HTTP `502`
  - 返回：`AppChatReverse: Chat failed, 429`
- `POST /v1/videos` 第 3 次 → HTTP `200`
  - `id=video_17c98454d92a410bba375605`
  - `object=video`
  - `status=completed`
  - `model=grok-imagine-1.0-video`
  - `seconds=6`
  - 返回 `url=https://grok.tap365.org/v1/files/video/users/e915e63c-75e6-4681-954e-15de41579c9a/generated/85b1d297-4c3f-41f9-bd03-c6bc7bd7c7ed/generated_video.mp4`
- 对返回视频 URL 执行 Range 请求：
  - HTTP `206`
  - `content-type=video/mp4`
  - `content-range=bytes 0-2047/1521330`
  - 本地 `file` 识别为 `ISO Media, MP4 Base Media v1 [ISO 14496-12:2003]`

#### sublb 公网 `sub-lb.tap365.org`
- `GET /v1/models` → HTTP `200`，返回 `data` 数量为 `3`。
- `POST /v1/videos` → HTTP `404`
  - 返回正文：`404 page not found`

### 10.8 结论
1. **上游 `grok.tap365.org` 的视频生成这次已经真实跑通**：`POST /v1/videos` 第三次返回 `status=completed`，并拿到可访问的 `video/mp4` 文件片段。
2. 前两次仍出现 `AppChatReverse: Chat failed, 429`，说明上游账号/内层池仍存在不稳定或被限流账号；但它已经不是“上游不支持视频”的问题。
3. **`sub-lb.tap365.org` 公网层本轮仍未跑通视频**：`POST /v1/videos` 返回 `404 page not found`。当前阻断点在 sublb 公网路由/部署版本/转发入口，不在上游视频能力本身。
4. 如果要对外公布 sublb 视频能力，必须先把 `/v1/videos` 正确挂到公网 sublb，并再次用 sublb key 完成 `POST /v1/videos` 业务接口实测；不能只凭上游成功宣布 sublb 已支持。

### 10.9 失败信息
- 上游失败样本：`502 upstream_error`，`AppChatReverse: Chat failed, 429`，第 1/2 次复现。
- sublb 失败样本：`404 page not found`，说明公网路由没有命中公开视频接口。
