# 第三方客户端问题汇总

> 最后整理：2026-03-30  
> 目标：专门回答“为什么同一个 sublb 网关，在不同客户端里表现不一样”。  
> 重点客户端：**OpenClaw、Codex CLI、curl、Python/OpenAI SDK**。  
> 阅读方式：先按客户端找，再看对应报错和解决动作。

---

## 0. 先写我自己的实战经验

这份文档不是泛泛而谈，是基于我自己本机的真实排障经验整理出来的。

### 0.0 我当前使用的 OpenClaw 版本

- `OpenClaw 2026.3.13 (61d171a)`

### 0.1 模型强度体感对照

- `claude-sonnet4.6` ≈ `gpt-5.4`，强度可以按 **medium** 理解
- `claude-opus4.6` ≈ `gpt-5.4`，强度可以按 **xhigh** 理解

这不是官方标定，而是我自己在真实使用里的体感映射，方便后面排第三方客户端时统一预期。

### 0.2 我的排障前提

我这边遇到第三方客户端异常时，**默认已经先排过环境**，不会一上来就把锅甩给客户端：

1. 先排本地网络环境
2. 先排代理环境变量污染
3. 先排 key / 订阅 / group
4. 先排 `/v1/models`
5. 先排 `/v1/responses`
6. 最后才回到具体客户端

所以如果文档里写“客户端问题”，很多时候真正意思是：

> **在基础环境已经先排过的前提下，这个客户端仍然有哪些特有误判点。**

## 1. 先讲结论：很多“客户端报错”根本不是客户端本体坏了

在 sublb 这类 OpenAI-compatible 网关上，第三方客户端报错通常来自 4 类问题：

1. **订阅 / key / group 问题**
2. **服务器侧配额或限流问题**
3. **上游 OpenAI / 模型路由问题**
4. **客户端协议形态与网关兼容性问题**

所以正确排障顺序永远是：

1. 先 `curl /v1/models`
2. 再 `curl /v1/responses`
3. 最后才回客户端

如果你跳过前两步，直接在客户端里猜，十有八九会猜错。

---

## 2. 所有第三方客户端通用的最小自检

```bash
export SUBLB_BASE_URL='https://sub-lb.tap365.org/v1'
export SUBLB_API_KEY='<你的 key>'

# 1) 看能不能进门
curl -sS \
  -H "Authorization: Bearer $SUBLB_API_KEY" \
  "$SUBLB_BASE_URL/models"

# 2) 看 gpt-5.4 能不能最小出字
curl -sS \
  -H "Authorization: Bearer $SUBLB_API_KEY" \
  -H 'Content-Type: application/json' \
  -d '{"model":"gpt-5.4","input":"Reply with exactly pong and nothing else."}' \
  "$SUBLB_BASE_URL/responses"
```

### 怎么看结果

- `/models` 403：优先查 key / 订阅 / group
- `/models` 200，但 `/responses` 失败：优先查模型路由 / 上游 / 协议兼容
- 两条都 200：再去看第三方客户端本身

## 3. OpenClaw 问题汇总（重点）

OpenClaw 是最容易让人误判的客户端，所以单独展开。

## 3.0 我本机 OpenClaw 的 tap365 示例配置（真实可抄）

我电脑上当前就是这样配的，**URL 不变**：

```json
{
  "models": {
    "providers": {
      "tap365": {
        "baseUrl": "https://sub-lb.tap365.org/v1",
        "apiKey": "<YOUR_API_KEY>",
        "api": "openai-responses",
        "models": [
          {
            "id": "gpt-5.4",
            "name": "GPT-5.4 via tap365",
            "input": ["text", "image"]
          }
        ]
      }
    }
  },
  "agents": {
    "defaults": {
      "model": {
        "primary": "tap365/gpt-5.4"
      },
      "models": {
        "tap365/gpt-5.4": {}
      }
    },
    "list": [
      {
        "id": "main",
        "model": "tap365/gpt-5.4"
      }
    ]
  }
}
```

### 这套配置当前的真实验证结果

- `GET https://sub-lb.tap365.org/v1/models` → `200`
- `POST https://sub-lb.tap365.org/v1/responses` + `model=gpt-5.4` → `200`
- 最小输入 `Reply with exactly pong and nothing else.` → 真实返回 `pong`

也就是说：

> **tap365 这家供应商当前本身是通的。**

如果 OpenClaw 这时仍然报错，就不要先怀疑 `sub-lb.tap365.org` 不可用，而要优先看 OpenClaw 本地运行态、依赖层、协议兼容和环境污染。

## 3.1 OpenClaw 报 `403 status code (no body)`

### 这通常不代表什么

它**不自动代表**：

- OpenClaw 装坏了
- OpenClaw 配置文件损坏
- OpenClaw 不支持 sublb

### 它更常代表什么

> OpenClaw 把上游真实 403 简化了，没有把原始 JSON 错误完整打印出来。

最常见根因：

- `SUBSCRIPTION_NOT_FOUND`
- key 对应 group 没有效订阅
- key 过期 / group 不允许

### 正确处理

1. 先绕过 OpenClaw 直打 `/models`
2. 再直打 `/responses`
3. 如果两条也 403，就先修 key / 订阅，不要继续折腾 OpenClaw

---

## 3.2 OpenClaw 报 `Your request was blocked`

### 这代表什么

> 更像是上游 WAF / Cloudflare / 风控拦截，不是 OpenClaw 自己的业务错误码。

### 常见场景

- provider 入口有 Cloudflare
- OpenClaw 当前请求形态被上游风控命中
- 代理环境变量污染，导致网络路径异常

### 正确处理

1. 先用同一 `baseUrl + key + model` 做 curl 直连
2. 如果 curl 正常、OpenClaw 被拦，检查 OpenClaw 进程是否继承了全局代理环境变量
3. 如果 curl 也被拦，优先怀疑 provider / Cloudflare，而不是 OpenClaw

---

## 3.3 OpenClaw 明明能列模型，但对话失败

### 这代表什么

这说明：

- OpenClaw 到网关入口是通的
- 但具体调用模型时失败了

### 常见根因

- 某个模型虽然可见，但实际不可用
- `/v1/responses` 与 OpenClaw 当前使用的协议组合不兼容
- 上游 502 / 429 / SSE 收尾异常

### 正确处理

1. 先测试 `gpt-5.4`
2. 再测试你在 OpenClaw 里选的那个模型
3. 如果 `gpt-5.4` 正常、目标模型不正常，按“模型粒度问题”处理

---

## 3.4 OpenClaw 用 `openai-responses` 不稳定，怎么办

### 现象

你可能会看到：

- `/v1/chat/completions` 能跑
- `/v1/responses` 在 OpenClaw 里表现异常
- 或 OpenClaw 对某些 provider 返回 400/403/no body

### 正确理解

这通常说明：

> 目标网关或上游对 Responses API 只是“半兼容”，不是完整兼容。

### 处理建议

- 如果你只是想先稳定跑通，优先选更稳定的协议入口
- 如果当前 provider 的 `openai-responses` 路径不稳定，可以考虑切到 `openai-completions` 路径验证
- 但前提仍然是：先把 curl 直连验证清楚

### 我这次在本机遇到的额外现实问题

这次我本机真实遇到的，不是 tap365 网关挂了，而是：

- `tap365` 直连测试已经通过；
- 但本地 `openclaw` CLI 启动时报：

```text
SyntaxError: The requested module '@mariozechner/pi-ai' does not provide an export named 'getOAuthApiKey'
```

这说明当前这份 OpenClaw 本地仓库还可能存在：

- `dist` 构建产物和依赖版本不匹配
- 或 `pnpm` 依赖链接层异常

所以一定要分清：

> **供应商可用，不等于你本地这份 OpenClaw 构建态没问题。**

---

## 3.5 OpenClaw 推荐排障顺序

```text
OpenClaw 报错
  ↓
先 curl /v1/models
  ↓
再 curl /v1/responses
  ↓
如果网关直连也失败：先修网关侧问题
  ↓
如果网关直连正常：再查 OpenClaw 配置、协议类型、代理环境变量
```

一句话：

> **OpenClaw 报错先查网关，不要先怪 OpenClaw。**

## 4. Codex CLI 问题汇总

## 4.1 `stream disconnected before completion`

### 这代表什么

> 流开始了，但客户端没有收到完整的结束事件。

### 常见根因

- `/v1/responses` 的 SSE 收尾不完整
- 代理链路截断
- 上游流式结束事件异常

### 正确处理

1. 先用 curl 看 `/v1/responses` 原始行为
2. 确认是不是所有请求都断流，还是只有某个模型断流
3. 再回头检查 Codex CLI

### 不要怎么做

- 不要一上来就先删配置重登
- 不要先假设是 Codex CLI 版本问题

---

## 4.2 Codex CLI 能登录，但调用失败

### 常见原因

- key 实际没有有效订阅
- 指定模型不可用
- 网关流式兼容性不稳定

### 正确处理

- 先单测 `gpt-5.4`
- 再单测你在 Codex 里配置的目标模型
- 如果 `gpt-5.4` 正常，说明不是整个 key 挂了

---

## 5. curl 用户常见问题

## 5.1 `error code: 502`

### 这代表什么

> 请求进了网关，但目标模型的转发链路失败。

### 怎么处理

- 先换 `gpt-5.4` 测试
- 如果只有某几个模型 502，就是模型粒度问题
- 如果全都 502，再看网关整体状态

---

## 5.2 `curl_exit=28`

### 这代表什么

> 超时。

### 怎么处理

- 增加 `--max-time`
- 换稳定模型测试
- 判断是不是只有个别模型慢

---

## 5.3 `curl_exit=35`

### 这代表什么

> TLS/SSL 握手失败。

### 怎么处理

- 换网络重试
- 先测别的模型
- 如果只在部分模型上出现，按上游链路问题处理

---

## 6. Python / OpenAI SDK 用户常见问题

## 6.1 SDK 报 429，但你分不清是谁的 429

### 先看错误内容

#### 情况 A：我们服务器自己的 429

通常会带结构化错误码，例如：

- `API_KEY_RATE_5H_EXCEEDED`
- `API_KEY_RATE_1D_EXCEEDED`
- `API_KEY_RATE_7D_EXCEEDED`
- `API_KEY_QUOTA_EXHAUSTED`
- `rate_limit_exceeded`

处理：

- 等时间窗恢复
- 换 key
- 升级额度

#### 情况 B：OpenAI 上游 429

典型文本：

```text
exceeded retry limit, last status: 429 Too Many Requests, request ...
```

处理：

> **请求速率过快，等待几分钟即可。**

不要继续狂重试。

---

## 6.2 SDK 报 JSON 解析错误 / invalid json response

### 这通常代表什么

- 你期待拿到普通 JSON
- 但上游 / 网关实际吐了 SSE 或非预期内容

### 怎么处理

- 优先使用非流式最小请求确认 `/responses` 是否真的返回标准 JSON
- 如果你的 SDK 期望 JSON，就不要把“上游内部 SSE”直接暴露给 SDK
- 先确认网关有没有做好 SSE → JSON 聚合

---

## 7. 第三方客户端最容易踩的误判

## 7.1 误判一：客户端报 403 = 客户端坏了

错。

很多 403 根因是：

- 订阅没了
- key 过期
- group 不允许
- OpenAI 上游或 Cloudflare 在拦

---

## 7.2 误判二：`/v1/models` 是 200 = 我的客户端配置一定没问题

错。

`/models` 只能证明：

- 入口通
- key 认证过了

不能证明：

- 你选的模型一定能跑
- Responses API 一定完整兼容
- 流式链路一定没问题

---

## 7.3 误判三：429 一律换 key

错。

如果是：

```text
exceeded retry limit, last status: 429 Too Many Requests, request ...
```

这更像 **OpenAI 上游限流**，第一动作是：

- 降速
- 等几分钟

不是立刻换客户端，不是继续猛撞。

---

## 8. 给 QA / 客服 / 接入方的标准回复模板

### 模板 1：OpenClaw 用户

> 你这个报错先不要直接认定是 OpenClaw 配坏了。请先用同一把 key 直接测试 `/v1/models` 和 `/v1/responses`。如果直连也失败，优先查订阅、group、模型路由；如果直连正常，再回头查 OpenClaw 的协议和代理配置。

### 模板 2：Codex CLI 用户

> `stream disconnected before completion` 通常不是账号登录失败，而是流式收尾异常。请先用 curl 验证 `/v1/responses` 的最小请求是否稳定，再决定是否继续查 Codex CLI。

### 模板 3：OpenAI SDK / Python 用户

> 如果你看到 `exceeded retry limit, last status: 429 Too Many Requests`，这通常是 OpenAI 上游限流，先等几分钟，不要高频重试。如果是带结构化 `reason` 的 429，则优先按服务器侧限流处理。

---

## 9. 最后收口

### 你只要记住这 6 句话

1. **第三方客户端报错，先查网关，不要先查客户端。**
2. **OpenClaw 的 403/no body 经常只是上游错误被简化了。**
3. **Codex CLI 的流断开，优先查 SSE 收尾。**
4. **`/v1/models` 200 不等于目标模型一定可用。**
5. **OpenAI 的 `429 Too Many Requests` 那句，通常是速率过快，等几分钟即可。**
6. **能不能列模型，和能不能稳定出字，是两回事。**
